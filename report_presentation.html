Bài toán Đặt ra & Mục tiêu

Bài toán thực tế

Thị trường ẩm thực tại Thành phố Hồ Chí Minh vô cùng đa dạng và cạnh tranh. Tuy nhiên, dữ liệu về đánh giá của khách hàng, mức độ phổ biến của các món ăn và chất lượng dịch vụ lại nằm rải rác và không có cấu trúc.


Yêu cầu: Cần một hệ thống thu thập, lưu trữ, và xử lý phân tán lượng lớn dữ liệu đánh giá này để trích xuất các insight có giá trị kinh doanh.

Mục tiêu Đồ án
  • Xây dựng pipeline Big Data hoàn chỉnh từ thu thập đến trực quan hóa.
  • Phân tích phân khúc nhà hàng theo đánh giá (1-5 sao) và khu vực địa lý (quận/huyện).
  • Phân tích cảm xúc (Sentiment Analysis) từ bình luận người dùng.
  • Kết hợp dữ liệu công thức nấu ăn quốc tế để tìm xu hướng ẩm thực.
  • Sử dụng hệ sinh thái Hadoop (HDFS, YARN, Hive, MapReduce).
Kiến trúc hệ thống

Pipeline Thu thập & Xử lý Dữ liệu Lớn

Kiến trúc Pipeline từ Crawler (TripAdvisor, TheMealDB) → Staging DB (MongoDB, MySQL) → ETL Sync → HDFS Storage → Phân tích (Hive, MapReduce) → Dashboard (Streamlit).

Hành trình chọn Công cụ Crawl

1
Thất bại với HTTP Requests truyền thống
Sử dụng thư viện `requests`, `curl_cffi` và `scrapy` thuần đều lập tức nhận mã lỗi HTTP 403 Forbidden. TripAdvisor dùng hệ thống DataDome/PerimeterX phát hiện không có trình duyệt thật (không thực thi được JS).
2
Giải pháp: Scrapy kết hợp Playwright
  • Scrapy: Framework thu thập dữ liệu mạnh mẽ, quản lý luồng (concurrency), tự động theo đuổi link (spider), pipelines.
  • Playwright: Công cụ tự động hóa trình duyệt (Chromium/Firefox/WebKit), render HTML như người dùng thật.
  • Sự kết hợp (scrapy-playwright): Scrapy điều phối logic lấy link, nhưng giao việc fetch HTML cho Playwright mở trình duyệt giả lập. Trang web tưởng đây là con người, tải toàn bộ React DOM và cấp quyền truy cập.

Kỹ thuật XPath và CSS Selector

Phân biệt CSS Selector & XPath

CSS SelectorCú pháp ngắn gọn, dễ đọc, tìm theo id (`#`), class (`.`), tag. Dùng tốt cho layout tĩnh. Không thể duyệt ngược lên cha.
XPathCú pháp phức tạp hơn nhưng sức mạnh tuyệt đối. Tìm theo nội dung text, duyệt ngược cây DOM, tìm phần tử theo cấu trúc logic phức tạp.

Áp dụng trong Đồ án

  • Dùng CSS Selector: Tìm vùng khoanh vùng đánh giá (VD: `div[data-automation="reviewCard"]`). Vì các thẻ này có thuộc tính rõ ràng.
  • Dùng XPath: Tìm Rating (`//svg/title[contains(text(), "bubbles")]`). TripAdvisor render rating bằng file SVG không có class cố định, bắt buộc phải dùng XPath để tìm chữ "bubbles" ẩn bên trong.
  • Step-by-step: Bật DevTools (F12) -> Inspect phần tử -> Lấy XPath tương đối -> Test trên console browser -> Đưa vào Scrapy.

Cài đặt MongoDB & Upsert Pipeline

🍃
Tại sao chọn MongoDB?
Dữ liệu cào về có cấu trúc không đồng nhất (nhà hàng có reviews, nhà hàng không có). MongoDB (NoSQL) lưu trữ dữ liệu dạng JSON/BSON, đặc biệt tốt cho cấu trúc mảng lồng nhau (Nested Array) như danh sách Reviews.
⚙️
Cơ chế Upsert (Update or Insert)
Scrapy được cấu hình Item Pipeline kết nối trực tiếp đến MongoDB. Khi cào được 1 nhà hàng:
  • Hệ thống lấy URL làm khóa chính `_id`.
  • Kiểm tra trong DB: Nếu URL đã tồn tại -> Cập nhật thông tin mới nhất.
  • Nếu URL chưa có -> Thêm mới bản ghi.
  • Đảm bảo chạy crawler nhiều lần không bao giờ sinh ra dữ liệu rác/trùng lặp.

Lỗi kết nối HTTP 403 Forbidden

Nguyên nhân
Mặc dù đã dùng Playwright Chromium, trình duyệt vẫn bị lộ cờ hiệu `navigator.webdriver = true`. Hệ thống DataDome nhận diện đây là bot tự động và lập tức chặn truy cập (403).
Giải pháp (Stealth Mode)
  • Thêm tham số `--disable-blink-features=AutomationControlled` vào Playwright.
  • Cài đặt thư viện `playwright-stealth` để xóa các dấu vết của bot.
  • Cung cấp `USER_AGENT` của trình duyệt Chrome thật trên Windows.
Kết quả: Qua mặt thành công, nhận mã 200 OK.

Mất địa chỉ (Null Address) khi sang trang

Nguyên nhân
Trang thông tin chính và các trang review tiếp theo của TripAdvisor có cấu trúc HTML khác nhau. Khi bot chạy lại lệnh XPath tìm địa chỉ trên trang số 2, XPath bị sai, trả về `null` và ghi đè mất địa chỉ đúng đã lấy được ở trang 1.

Dữ liệu bẩn (Dirty Data)

Nguyên nhân
Dữ liệu cào về có dạng text thô, không khớp chuẩn phân tích:

- Số lượng đánh giá: `"(112)"` (chuỗi text)
- Điểm sao: `"5 of 5 bubbles"`
- Địa chỉ bị dính Zipcode: `"Ho Chi Minh City 70000"`
Giải pháp (Data Normalization)
  • Xây dựng script Python (`init_db.py`) chạy sau bước cào.
  • Dùng Regular Expression trích xuất số: `"(112)"` -> `112` (Integer).
  • Regex lấy điểm: `"5 of 5 bubbles"` -> `5.0` (Float).
  • Xử lý tách chính xác tên "Quận 1", "Quận 3" từ chuỗi địa chỉ dài.

Tối ưu số lần gọi API

Thay vì gọi từng ID món ăn, hệ thống gọi API endpoint `search.php?f={letter}` duyệt từ A đến Z. Chỉ với 26 requests, thu thập được toàn bộ ~300+ món ăn miễn phí, không bao giờ vi phạm Rate Limit.

API trả về 20 cột riêng biệt: `strIngredient1` đến `strIngredient20`. Điều này cực kỳ khó cho MapReduce.

  • Dùng vòng lặp động trong Python gom 20 cột này thành 1 mảng JSON duy nhất: `ingredients: [...]`
  • Loại bỏ các nguyên liệu rỗng hoặc Null.
  • Insert trực tiếp Collection `meals` vào MongoDB.

Cấu trúc Dữ liệu

Thiết kế & Chuẩn hóa MySQL Schema

Tách bảng Relational DB

Dữ liệu thô từ MongoDB (Nested Documents) không phù hợp cho truy vấn Streamlit hay MapReduce phân tích theo chiều dọc. Script `init_db.py` thực hiện phân rã thành các bảng chuẩn:

  • Bảng `restaurants`: `id` (PK), `name`, `rating`, `review_count`, `district_parsed`, `city`.
  • Bảng `reviews`: `id` (PK), `restaurant_id` (FK), `user`, `rating`, `comment`. Hơn 44,000 dòng.
  • Bảng `meals`: `id` (PK), `name`, `category`, `area`, `ingredients` (dạng text).

Cấu hình MySQL & Tối ưu

  • Cấu hình MySQL tự động chạy ngầm trên Ubuntu WSL2.
  • Tạo Dedicated Database `food_sentiment_db`.
  • Sử dụng cấu trúc Khóa chính (Primary Key) và Khóa ngoại (Foreign Key) đảm bảo tính toàn vẹn dữ liệu.
  • Các thao tác Thêm, Xóa, Sửa (CRUD) từ giao diện Streamlit sẽ phản hồi trực tiếp lên Schema MySQL này một cách chuẩn xác.

Hadoop Infrastructure & Data Sync

Cầu nối OLTP & OLAP

1. Đồng bộ lên HDFS

Trích xuất toàn bộ dữ liệu từ MongoDB & MySQL thành các file phân tán JSONL, tự động upload lên Hệ thống lưu trữ Hadoop (HDFS) an toàn.

Sử dụng Apache Hive mapping dữ liệu thô trên HDFS thành các External Tables. Chuyển đổi dữ liệu NoSQL phức tạp (Nested Arrays) sang bảng SQL dễ truy vấn.

Data Ingestion

Luồng đồng bộ dữ liệu (MySQL / MongoDB → HDFS)

Thách thức của Dữ liệu thô

  • Cơ sở dữ liệu trung gian (Staging DB) không thích hợp để MapReduce phân tích trực tiếp.

  • Đẩy thẳng lên directory `/data/raw/` của HDFS với Replication Factor=3.

Lỗi không thể đồng bộ Decimal MySQL

Nguyên nhân
Script đẩy MySQL sang HDFS bị sập lỗi `TypeError: Object of type Decimal is not JSON serializable`. Module JSON mặc định của Python không hiểu được kiểu dữ liệu thập phân (`Decimal`) của MySQL.
Giải pháp
  • Viết đè (Override) một lớp `MySQLJSONEncoder` tùy chỉnh trong Python kế thừa từ `json.JSONEncoder`.
  • Xử lý ngoại lệ: Nếu gặp kiểu `Decimal`, ép kiểu thủ công về `float` trước khi parse thành JSON. Script chạy mượt mà.
Hadoop · Vấn đề 2 / 6

Lỗi mã hóa ObjectId MongoDB

Nguyên nhân
Tương tự MySQL, trường `_id` của MongoDB sử dụng định dạng nhị phân nội bộ là `ObjectId`. Hive và JSON chuẩn hoàn toàn không thể biên dịch và đọc được loại dữ liệu nhị phân nguyên thủy này.
Giải pháp
  • Tạo `MongoJSONEncoder` để chuyển đổi `ObjectId` thành một chuỗi `String` bình thường (`str(o)`).
  • Cấu trúc lại khóa: Chuyển tên khóa `_id` của MongoDB thành `id` thông thường để Hive ánh xạ tên cột dễ dàng hơn.

HDFS Directory Already Exists

Nguyên nhân
Hadoop HDFS được thiết kế để bảo vệ dữ liệu. Lệnh `hdfs dfs -put` sẽ báo lỗi và từ chối chạy nếu thư mục hoặc file đích đã tồn tại trên HDFS từ các lần chạy trước, gây cản trở việc tự động hóa.
Giải pháp
  • Bổ sung tham số `-f` (Force) vào lệnh put: `hdfs dfs -put -f `.
  • Tham số này yêu cầu Hadoop tự động ghi đè dữ liệu cũ bằng dữ liệu mới mà không hiển thị lỗi. Pipeline trở nên tự động hoàn toàn (idempotent).

Lỗi Location không tìm thấy của Hive

Nguyên nhân
Khi tạo External Table trong Hive và chỉ định `LOCATION '/data/raw/restaurants/restaurants.jsonl'`, Hive báo lỗi không tìm thấy dữ liệu. Mặc dù file đã tồn tại trên HDFS.
Giải pháp
  • Cơ chế của Hive là quét thư mục (Directory) chứ không ánh xạ trực tiếp đến một file cụ thể.
  • Thay đổi câu lệnh thành `LOCATION '/data/raw/restaurants/'`. Hive sẽ tự động scan và đọc tất cả các file JSONL nằm trong thư mục đó.

Nguyên nhân

Parse Mảng Dữ Liệu Lồng Nhau (Nested Arrays)

Nguyên nhân
Một nhà hàng có danh sách rất nhiều Review chứa bên trong (User, Rating, Comment). Hive Query thông thường `SELECT reviews FROM...` trả về chuỗi vô định dạng hoặc báo NULL vì không hiểu được mảng đối tượng.
Giải pháp

Crash do lỗi định dạng JSON

Nguyên nhân
Bất kỳ dòng text nào bị sai lệch định dạng (dấu nháy hỏng, ký tự UTF-8 không hợp lệ) từ HDFS đều khiến quá trình `json.loads(line)` của hàm Mapper bị văng lỗi (JSONDecodeError), làm sập toàn bộ Job Hadoop đang chạy.
Giải pháp
  • Triển khai cơ chế bẫy lỗi try...except (ValueError, JSONDecodeError) xung quanh khối parse.
  • Hadoop tự động bỏ qua (skip silently) dòng dữ liệu rác đó và tiếp tục phân tích hàng vạn dòng hợp lệ còn lại mà không bị gián đoạn.
MapReduce · Vấn đề 2 / 6

Treo RAM do Regex (Ingredient Match)

Nguyên nhân
Thuật toán dò tìm 300+ tên nguyên liệu trong nội dung văn bản. Nếu biên dịch Regex 1 cách ngây ngô, Engine Regex sẽ bị bùng nổ không gian trạng thái (State Explosion). Hậu quả: MapReduce bị treo, tràn RAM (OOM).
Giải pháp
  • Sử dụng hàm mapper_init() để biên dịch (Compile) regex pattern đúng 1 lần duy nhất lúc khởi động Node.
  • Sắp xếp danh sách 300 từ khóa theo chiều dài giảm dần (Length DESC), giúp Regex ưu tiên cụm từ dài trước, tối ưu tốc độ vượt bậc.
MapReduce · Vấn đề 3 / 6

Lỗi sai số thập phân (Floating Point)

Nguyên nhân
Tính trung bình Rating (Average) trên Python sinh ra các số vô hạn như `4.3450000001`. Khi cộng dồn qua nhiều node phân tán, hệ thống gặp sai số Rounding Error. Kết quả khác với truy vấn SQL của Hive.
Giải pháp
  • Can thiệp thẳng vào hàm Reducer cuối cùng.
  • Sau khi tổng hợp, áp dụng hàm `round(avg, 2)` ép chuẩn kết quả về 2 chữ số thập phân trước khi xuất ra output JSON. Đảm bảo 100% khớp với Hive.

Lệch điểm Sentiment Analysis

Nguyên nhân
Khách hàng viết review bằng cả chữ hoa, chữ thường, hoặc viết HOA toàn bộ (ví dụ: "GOOD", "good", "GoOd"). Nhưng từ điển phân tích từ tích cực (Lexicon) chỉ có chữ thường. Việc so khớp bị trượt, làm điểm số bị thấp hơn thực tế.
Giải pháp

  • Việc Case-Insensitive Matching giúp MapReduce bắt được chính xác 100% mọi từ khóa tích cực/tiêu cực.

Lỗi từ chối kết nối MySQL (WSL2)

Nguyên nhân
Môi trường giả lập WSL2 và Windows Host có lớp mạng riêng biệt. Khi kết nối MySQL với tham số Host là `localhost`, Streamlit bị bối rối và sinh ra lỗi `Connection refused`.

Mất kết nối Hive (HiveServer2 Timeout)

Nguyên nhân

Giải pháp (3-Layer Fallback)
  • Tạo hàm kết nối bảo vệ 3 lớp:
  • Tầng 1: Ưu tiên dùng PyHive TCP/IP (Nhanh nhất).
  • Tầng 2: Nếu lỗi, giáng cấp gọi lệnh Subprocess (`hive -S -e "SQL"`).
  • Tầng 3: Nếu vẫn lỗi mạng, load `Mock Data` (Dữ liệu dự phòng) để UI luôn luôn vẽ được biểu đồ không bao giờ sập.
